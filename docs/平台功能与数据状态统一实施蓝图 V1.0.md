## 1. 蓝图的使用方式

### 1.1 三层事实源

1. 上位规范定义技术栈、工程、安全和质量红线。
2. 本蓝图定义产品功能、应用用例、业务规则和数据状态。
3. 机器清单定义可核查的功能 ID 及其端到端制品映射。

三者缺一不可。AI 不得只根据页面原型开发，也不得只根据数据库表生成 CRUD。

### 1.2 “一个功能完整”的判定

每个功能 ID 必须同时交付：

```text
用户场景
  + 页面/交互
  + 权限与资源范围
  + REST/WebSocket/MCP/事件契约
  + 后端 Command/Query 用例
  + 领域规则与状态迁移
  + 数据迁移和所有权
  + 审计/Trace/指标/告警
  + 单元/契约/集成/E2E 测试
  + 部署/升级/回滚与安全扫描
  + 运行和故障处理说明
```

缺少其中任一项时，只能标记 `IN_PROGRESS`，不得标记 `DONE`。

### 1.3 实施追踪字段

需求、分支、提交、测试计划和发布记录统一引用：

- 功能 ID：`F-<DOMAIN>-<NNN>`。
- 页面 ID：`P-<DOMAIN>-<NNN>`。
- 用例 ID：`UC-<DOMAIN>-<NNN>`。
- 状态模型 ID：`SM-<DOMAIN>-<NNN>`。
- 验收 ID：上位规范的 `C01-C24`。
- 公共能力复用：`platform_capability_profile`。
- 运行目标：`deployment_targets`。
- 外部契约：`EXT-<COMPONENT>`，使用前必须 `PINNED`。

新增 ID 先更新本蓝图和机器清单；禁止在代码中出现无来源的私有流程。

清单顶层 `architecture`、`platform_capabilities`、`platform_capability_profiles`、`external_contracts` 和 `mandatory_done_evidence` 适用于每个功能，不能只读取 `features` 子项。公共能力唯一归属变更、技术栈变更或 React 例外必须有 ADR。

## 2. 产品信息架构

### 2.1 前端主导航

| 导航 | 页面范围 | 默认可见角色 |
| --- | --- | --- |
| 工作台 | 运行总览、任务态势、告警摘要 | Observer 及以上 |
| 设备中心 | 机器人、型号、分组、网关、传感器 | Observer、Maintainer、Admin |
| 任务中心 | 任务、创建、模板、审批、调度 | Dispatcher、Approver、Admin |
| 能力中心 | Capability、Skill、版本、MCP Tool | Maintainer、Admin |
| 地图场景 | 地图、场景、区域、禁区、风险点 | Dispatcher、Maintainer、Admin |
| 实时监控 | 地图态势、视频、人工接管、Trace 回放 | Dispatcher、Observer、Auditor |
| 告警运维 | 告警、服务健康、边缘状态、OTA | Maintainer、Admin |
| 记忆优化 | 任务记忆、失败案例、改进候选、评测 | Maintainer、Approver |
| 系统管理 | 用户、组织、角色、权限、字典、国际化、配置、审计 | Admin、Auditor |

角色只决定默认授权集合。前端和后端实际判断稳定权限码，不判断角色名称。

### 2.2 全局前端布局

- 顶栏：环境标识、云边总体状态、当前组织、语言、时区、用户菜单。
- 左侧导航：按权限动态显示；隐藏菜单不能代替后端鉴权。
- 全局任务抽屉：展示当前用户可见的运行任务、告警和人工接管租约。
- 全局急停入口：只对有权限用户显示，固定可达，不藏入多层菜单；触发对象和影响范围必须清晰。
- 主内容区：面包屑、页面标题、数据时间戳、操作区、内容、帮助/Runbook。
- 断网提示：明确区分“浏览器到云断连”“云到边缘断连”“机器人离线”。

### 2.3 页面通用状态

页面视图状态与领域状态分离，统一为：

```text
INITIAL
LOADING
READY
EMPTY
PARTIAL
STALE
OFFLINE
FORBIDDEN
ERROR
```

规则：

- `PARTIAL`：部分组件/数据源失败，页面仍可用；必须指出缺失内容。
- `STALE`：有缓存快照但实时连接中断；显示最后更新时间，禁止当作实时数据。
- `OFFLINE`：浏览器无法连接云端；只允许展示已缓存只读数据，不接受控制命令。
- `FORBIDDEN`：已认证但无权限；不得伪装成 404，资源安全策略要求隐藏时除外。
- 命令提交成功只表示 `ACCEPTED`，不能立即在 UI 伪造最终领域状态。

### 2.4 列表页统一规则

- URL 保存筛选、排序、分页和视图模式，刷新可恢复。
- 服务端分页、排序和过滤；禁止拉全量后前端筛选。
- 默认显示数据时间戳和数据来源。
- 批量操作先预检权限和资源范围，逐项返回结果；不得全有或全无地吞掉细节。
- 空状态区分“没有数据”和“筛选无结果”。
- 导出是受审计的异步任务，不能让浏览器直接遍历全部分页。
- 表格字段、字典 label、时间、单位和状态标签使用统一组件。

### 2.5 详情页统一规则

- 顶部展示公共 ID、核心状态维度、更新时间和可执行动作。
- 使用摘要、配置、事件、审计、关联对象等标签页。
- 状态历史来自事实事件，不根据 `updated_at` 猜测。
- 变更操作带 `expected_version`/ETag；冲突时刷新并提示，不覆盖他人修改。
- 危险操作显示对象、影响、当前状态、策略结果和确认要求。

## 3. 前端页面与功能设计

### 3.1 登录与工作台

| 页面 ID | 路由 | 必须功能 | 主要权限 | 数据/实时来源 |
| --- | --- | --- | --- | --- |
| `P-AUTH-001` | `/login` | 登录、锁定提示、语言选择、会话恢复 | 匿名 | Auth REST |
| `P-AUTH-002` | `/profile` | 资料、语言、时区、活跃会话、退出其他会话 | 当前用户 | IAM REST |
| `P-DASH-001` | `/dashboard` | 设备在线、任务状态、告警、安全判定、系统健康摘要 | `dashboard.view` | REST 快照 + WebSocket |

工作台必须：

- 显示机器人总数、在线/离线、可调度、故障、急停数量。
- 显示任务各状态、近期开启/完成/失败和等待审批。
- 显示未解决告警及严重级别。
- 显示 Safety Gateway 拦截/修改趋势。
- 显示数据更新时间和每个组件的数据源健康。
- 卡片可跳转到带相同筛选条件的列表页。

工作台不得直接提供未经过上下文确认的高危控制；急停走独立受控流程。

### 3.2 设备中心

| 页面 ID | 路由 | 必须功能 | 主要权限 |
| --- | --- | --- | --- |
| `P-ROBOT-001` | `/robots` | 搜索、按组织/分组/型号/能力/多维状态筛选、批量标签 | `robot.robot.read` |
| `P-ROBOT-002` | `/robots/:robot_id` | 摘要、能力、传感器、状态、遥测、任务、告警、Trace、维护 | `robot.robot.read` |
| `P-ROBOT-003` | `/robot-models` | 型号、厂商、协议、兼容矩阵、安全模板 | `robot.model.read/manage` |
| `P-ROBOT-004` | `/robot-groups` | 分组、组织归属、地图/策略绑定 | `robot.group.manage` |
| `P-EDGE-001` | `/edge-gateways` | 网关身份、证书、连接、运行时、绑定机器人 | `edge.gateway.read` |
| `P-EDGE-002` | `/edge-gateways/:gateway_id` | 版本、适配器、队列、策略缓存、证书轮换、诊断 | `edge.gateway.read/manage` |

机器人详情必须将状态拆开展示：

- 注册状态。
- 连接状态。
- 运行状态。
- 安全状态。
- 维护状态。
- 电量、位姿、网络、传感器新鲜度。

禁止合并成模糊的“正常/异常”。“可调度”是后端根据多维状态计算的派生属性，并展示不可调度原因。

设备管理动作：

- 注册审批/激活/停用。
- 分组、地图、策略绑定。
- 进入/退出维护。
- 证书轮换/撤销。
- 查看 Capability Manifest 差异。
- 发起诊断；不得从云端执行任意 Shell。

每个动作均需权限、版本、原因、审计和结果反馈。

### 3.3 任务中心

| 页面 ID | 路由 | 必须功能 | 主要权限 |
| --- | --- | --- | --- |
| `P-MISSION-001` | `/missions` | 状态/风险/来源/机器人/时间筛选，任务列表和导出 | `mission.mission.read` |
| `P-MISSION-002` | `/missions/create` | 自然语言、结构化、模板三种创建方式 | `mission.mission.create` |
| `P-MISSION-003` | `/missions/:mission_id` | 计划 DAG、执行进度、步骤、策略、媒体、Trace、控制 | `mission.mission.read` |
| `P-MISSION-004` | `/mission-templates` | 模板版本、参数 Schema、风险和发布 | `mission.template.manage` |
| `P-MISSION-005` | `/mission-approvals` | 待审批、风险差异、计划/策略快照、批准/拒绝 | `mission.mission.approve` |
| `P-MISSION-006` | `/fleet-schedule` | 候选评分、资源占用、冲突、分配和重调度记录 | `fleet.schedule.read/manage` |

任务创建页必须：

- 自然语言模式保留原始命令，展示 Agent 生成的结构化提案。
- 结构化模式按机器人/区域/Skill/安全参数生成同一 Plan Schema。
- 模板模式选择已发布模板版本并填充参数。
- 在提交前显示目标、必要能力、风险级别、预计步骤和审批要求。
- 信息不足时要求补充，不能让 Agent 自行创造机器人、地图或区域。
- 创建请求有幂等键，重复提交只产生一个 Mission。

任务详情必须：

- 以 DAG 和时间轴两种方式展示步骤。
- 区分 Agent 提案、Planner 最终计划和审批后计划。
- 展示每次调度的候选、排除原因和评分。
- 展示云端 Policy 与边缘 Safety 的独立判定。
- 暂停、恢复、取消按钮由后端返回的 `available_actions` 决定。
- 急停不作为普通 Mission 状态按钮复用，走机器人/任务范围急停流程。
- 终态任务只读，不能通过前端“重新打开”；重试创建关联的新任务。

审批页必须：

- 显示发起人、计划摘要、风险、机器人、区域、时间、策略和仿真证据。
- 显示与上次修订的差异。
- 批准/拒绝需意见，高风险任务遵守职责分离。
- 计划、机器人、地图或策略版本变化后，旧审批自动失效。

### 3.4 能力中心

| 页面 ID | 路由 | 必须功能 | 主要权限 |
| --- | --- | --- | --- |
| `P-CAP-001` | `/capabilities` | 能力代码、版本、提供机器人、Schema | `skill.capability.read` |
| `P-SKILL-001` | `/skills` | 分类、风险、状态、版本、兼容设备筛选 | `skill.skill.read` |
| `P-SKILL-002` | `/skills/:skill_id` | 版本、输入/输出、依赖、安全约束、制品、测试证据 | `skill.skill.read/manage` |
| `P-MCP-001` | `/mcp-tools` | Tool Schema、权限、风险、灰度、调用统计 | `mcp.tool.read/manage` |

Skill 发布必须使用向导：

```text
创建草稿
-> 编辑 Schema/依赖/安全约束
-> 上传或关联签名制品
-> 契约测试
-> 仿真测试
-> 安全评审
-> 批准
-> 发布/灰度
```

- 已发布版本禁止编辑，只能复制为新草稿。
- 页面必须显示版本差异和兼容性判定。
- Tool 页面必须明确它映射到哪个应用用例，禁止出现 Mapper、ROS topic 或 SDK 入口。
- Tool 灰度按用户/组织/环境配置，不能跳过权限或 Safety。

### 3.5 地图与场景

| 页面 ID | 路由 | 必须功能 | 主要权限 |
| --- | --- | --- | --- |
| `P-MAP-001` | `/maps` | 地图资产、版本、坐标系、状态、绑定设备 | `map.map.read` |
| `P-MAP-002` | `/maps/:map_id` | 地图查看、版本、校验、发布和回滚 | `map.map.read/manage` |
| `P-SCENE-001` | `/scenes/:scene_id` | 区域、路径、停靠点、充电点、风险点编辑 | `map.scene.manage` |

地图编辑要求：

- 所有坐标显示 frame 和单位。
- 编辑产生草稿版本；发布后不可原地修改。
- 禁区变化触发策略缓存更新和受影响任务重评估。
- 地图上传、校验、发布是异步流程，页面显示每阶段结果。
- 运行任务绑定明确版本，页面不能只显示“当前地图”。

### 3.6 实时监控、媒体和接管

| 页面 ID | 路由 | 必须功能 | 主要权限 |
| --- | --- | --- | --- |
| `P-MON-001` | `/monitor/fleet` | 多机器人地图、状态、轨迹、任务、告警叠加 | `monitor.fleet.view` |
| `P-MON-002` | `/monitor/robots/:robot_id` | 实时遥测、视频、传感器、当前执行 | `monitor.robot.view` |
| `P-CONTROL-001` | `/control/:robot_id` | 获取控制租约、受控高层动作、释放租约 | `robot.robot.control` |
| `P-MEDIA-001` | `/media` | 图片/视频/地图/bag/日志包查询、预览、下载 | `media.asset.read` |
| `P-TRACE-001` | `/traces/:trace_id` | 决策链、事件时间线、参数、安全结果和回放 | `trace.trace.read` |

实时监控必须：

- 先取 REST 快照，再订阅 WebSocket 增量。
- 显示连接质量、数据延迟和最后事件游标。
- 乱序事件按聚合版本处理，重复事件去重。
- 断线后进入 `STALE`，禁止保留可用的控制按钮。

人工接管必须：

- 明示当前任务、影响和租约有效期。
- 获取租约成功后才显示动作控件。
- 只暴露注册的高层动作和参数边界，不提供裸速度/topic/SDK 控制。
- 所有动作仍经过 Safety Gateway。
- 租约失效、切页、断线或窗口关闭时自动请求释放并由边缘超时兜底。

Trace 回放必须区分：

- 事实回放：按已记录事件呈现，任何时候可用。
- 仿真重放：在隔离仿真环境重新执行，必须明确不是历史事实。
- 禁止“回放”直接向真实机器人重发历史动作。

### 3.7 告警与运维

| 页面 ID | 路由 | 必须功能 | 主要权限 |
| --- | --- | --- | --- |
| `P-ALARM-001` | `/alarms` | 告警筛选、确认、解决、抑制、关联对象 | `ops.alarm.read/manage` |
| `P-OPS-001` | `/ops/health` | 云服务、NATS、DB、S3、可观测栈、边缘健康 | `ops.health.read` |
| `P-OTA-001` | `/ops/ota` | 制品、兼容性、分批计划、进度、回滚 | `ops.ota.manage` |

告警页面：

- `ACKNOWLEDGED` 不得显示为已解决。
- 抑制必须填写原因和截止时间。
- 急停告警不能在机器人仍锁存时手工解决。
- 告警详情链接到 Mission、Robot、Gateway、Trace 和 Runbook。

OTA 页面：

- 发布制品签名和 SBOM 必须可见。
- 选择设备时校验型号、架构、系统和当前运行状态。
- 分批/暂停/回滚是后端受控状态机，不由前端循环发单设备命令。
- Safety/急停相关组件升级必须安排设备安全停机。

| 页面 ID | 路由 | 必须功能 | 主要权限 |

- 被纠正/失效的记忆保留历史，但默认检索排除。

| `P-SYS-I18N` | `/system/i18n` | 运行时资源、缺失 key、导入/导出 | `platform.i18n.manage` |
| `P-SYS-CONFIG` | `/system/configuration` | 类型化配置、作用域、版本、回滚 | `platform.config.manage` |
| `P-SYS-AUDIT` | `/system/audit` | 操作、主体、资源、结果、Trace 检索和导出 | `audit.audit.read` |

统一要求：

- 用户、角色、字典等管理页面不得在各业务模块重复建设。
- 权限目录由后端声明和迁移产生，UI 不允许任意创建代码不存在的权限。
- 修改角色权限必须显示新增/移除差异并使撤权及时生效。
- 字典页不得编辑状态机、安全决策、权限码、事件类型。
- i18n 页只管理运行时业务资源；代码静态文案仍随 Git 发布。
- 配置值按 Schema 编辑；Secret 只显示引用和轮换状态。
- 审计表只读，任何“删除日志”功能禁止出现。

## 4. 前端实现约束

### 4.1 后端驱动动作

详情接口统一返回：

```json
{
  "resource": {},
  "available_actions": [
    {
      "action": "MISSION_PAUSE",
      "permission": "mission.mission.pause",
      "enabled": true,
      "disabled_reason_code": null,
      "confirmation": "STANDARD"
    }
  ]
}
```

- 前端根据 `available_actions` 渲染动作，但后端执行时必须重新校验。
- 禁止前端复制完整状态迁移规则并自行推断可操作性。
- `disabled_reason_code` 使用 i18n 显示原因。

### 4.2 前端领域 Store

Pinia Store 按领域划分：

```text
authStore
platformStore
robotStore
missionStore
skillStore
mapStore
monitorStore
alarmStore
traceStore
```

- Store 保存服务端快照、版本和连接状态，不重新实现领域实体。
- WebSocket 事件通过统一 Event Reconciler 校验 `event_id`、聚合 ID 和版本后更新 Store。
- 页面卸载必须释放订阅和临时资源，控制租约由独立守护逻辑管理。

### 4.3 表单与命令

- 表单 Schema 尽量从 OpenAPI/JSON Schema 生成。
- 所有命令按钮防重复提交，但真正幂等由后端保证。
- 后端 409 版本冲突显示差异和刷新选项，不自动覆盖。
- 后端 202 表示异步受理，前端跟踪 Operation/Mission，不显示“已完成”。
- 后端 422 显示字段与领域错误；403 显示权限/范围不足；503 显示依赖不可用。

### 4.4 前端测试

每个页面至少覆盖：

- 有权限/无权限。
- Loading/Empty/Ready/Partial/Stale/Error。
- 字典和中英文。
- 命令成功、校验失败、409 冲突、202 异步、重复点击。
- WebSocket 重复、乱序、断线和重连。
- 适用的状态动作矩阵。

## 5. 后端应用用例统一设计

### 5.1 Command、Query 和异步 Operation

后端用例分为：

- Command：改变业务事实，必须带 `command_id`、主体、权限上下文、幂等键和期望版本。
- Query：读取事实或投影，不产生业务副作用。
- Operation：持续时间较长的异步工作，如规划、地图处理、导出、OTA。

Command 基本信封：

```json
{
  "command_id": "01...",
  "command_type": "mission.pause",
  "issued_at": "2026-07-03T08:00:00Z",
  "actor": {
    "type": "USER",
    "id": "user-001"
  },
  "resource_id": "mission-001",
  "expected_version": 7,
  "reason": "现场临时封控",
  "payload": {}
}
```

Command Handler 固定执行顺序：

```text
认证
-> 权限码
-> 数据范围
-> 幂等检查
-> 加载聚合和版本
-> 领域不变量/状态迁移
-> 同事务保存事实、审计、Outbox
-> 提交
-> 异步发布事件
```

外部 Agent、对象存储、NATS、边缘设备和模型调用不得处于数据库长事务内。

## 6. 统一数据状态模型

### 6.1 状态设计总则

- 禁止使用一个通用 `status` 混合注册、连接、运行、安全和健康语义。
- 状态字段必须对应一个明确状态机 ID，并在契约、数据库和代码中使用同一枚举。
- 状态码不是数据字典；显示 label 可通过 i18n。
- 每次迁移记录 `from_state`、`to_state`、原因码、主体、时间、聚合版本和 `trace_id`。
- 只有指定领域用例能迁移状态；Controller、Mapper、定时脚本不能直接改状态。
- 派生状态不作为可人工修改事实保存；必须能从源事实重算。
- 未知状态显式为 `UNKNOWN`，不得默认成正常。

### 6.2 用户状态 `SM-IAM-001`

```text
PENDING -> ACTIVE
PENDING -> DISABLED
ACTIVE -> LOCKED | DISABLED
LOCKED -> ACTIVE | DISABLED
DISABLED -> ACTIVE
```

| 状态 | 语义 |
| --- | --- |
| `PENDING` | 已创建，尚未完成初始激活 |
| `ACTIVE` | 可认证并按授权操作 |
| `LOCKED` | 因安全策略暂时禁止认证 |
| `DISABLED` | 管理停用，会话全部撤销 |

恢复 `DISABLED -> ACTIVE` 必须有管理权限和审计；删除用户不作为普通状态迁移。

### 6.3 Edge Gateway 多维状态

#### 身份状态 `SM-EDGE-001`

```text
PENDING -> ACTIVE -> SUSPENDED -> ACTIVE
PENDING | ACTIVE | SUSPENDED -> REVOKED
```

`REVOKED` 为终态，同一硬件重新接入需新身份。

#### 连接状态 `SM-EDGE-002`

```text
UNKNOWN -> ONLINE
ONLINE -> DEGRADED | OFFLINE
DEGRADED -> ONLINE | OFFLINE
OFFLINE -> ONLINE
```

连接状态由心跳租约和链路质量计算，不能人工设置。

#### 运行时状态 `SM-EDGE-003`

```text
STARTING -> READY | DEGRADED | FAILED
READY -> DEGRADED | FAILED
DEGRADED -> READY | FAILED
FAILED -> STARTING
```

只有 `identity=ACTIVE`、`connection=ONLINE/允许的 DEGRADED`、`runtime=READY` 且策略缓存有效时可接收普通执行命令。急停通道不受普通 readiness 阻断。

### 6.4 Robot 多维状态

#### 注册状态 `SM-ROBOT-001`

```text
PENDING -> ACTIVE
ACTIVE -> SUSPENDED -> ACTIVE
PENDING | ACTIVE | SUSPENDED -> REVOKED
```

#### 连接状态 `SM-ROBOT-002`

```text
UNKNOWN -> ONLINE
ONLINE -> DEGRADED | OFFLINE
DEGRADED -> ONLINE | OFFLINE
OFFLINE -> ONLINE
```

#### 运行状态 `SM-ROBOT-003`

```text
IDLE -> RESERVED -> EXECUTING
RESERVED -> IDLE | EXECUTING
EXECUTING -> PAUSED | IDLE | FAULT
PAUSED -> EXECUTING | IDLE | FAULT
IDLE -> MAINTENANCE
MAINTENANCE -> IDLE | FAULT
FAULT -> MAINTENANCE | IDLE
```

`FAULT -> IDLE` 需要故障已清除和健康自检；不得由普通任务完成事件直接迁移。

#### 安全状态 `SM-ROBOT-004`

```text
UNKNOWN -> NORMAL | RESTRICTED | ESTOPPED
NORMAL -> RESTRICTED | ESTOPPED
RESTRICTED -> NORMAL | ESTOPPED
ESTOPPED -> RESTRICTED
RESTRICTED -> NORMAL
```

急停复位先 `ESTOPPED -> RESTRICTED`，完成姿态/环境/设备自检后才可 `RESTRICTED -> NORMAL`。

#### 可调度派生规则

```text
schedulable =
  registration_state == ACTIVE
  AND connection_state == ONLINE
  AND operation_state == IDLE
  AND safety_state == NORMAL
  AND gateway.runtime_state == READY
  AND manifest_valid == true
  AND policy_cache_valid == true
  AND no_blocking_alarm
  AND required_capabilities_match
```

允许弱网 `DEGRADED` 调度只能由任务 Policy 显式放开，不能修改全局公式掩盖风险。

### 6.5 Mission 状态 `SM-MISSION-001`

```text
CREATED -> PLANNING
CREATED -> CANCELLED

PLANNING -> WAITING_APPROVAL | SCHEDULED | FAILED | CANCELLED
WAITING_APPROVAL -> SCHEDULED | CANCELLED | FAILED
SCHEDULED -> DISPATCHED | CANCELLED | FAILED
DISPATCHED -> RUNNING | FAILED | CANCELLED | EMERGENCY_STOPPED
RUNNING -> PAUSED | COMPLETED | FAILED | CANCELLED | EMERGENCY_STOPPED
PAUSED -> RUNNING | FAILED | CANCELLED | EMERGENCY_STOPPED
```

终态：

```text
COMPLETED
FAILED
CANCELLED
EMERGENCY_STOPPED
```

| 迁移 | 唯一触发事实 |
| --- | --- |
| `CREATED -> PLANNING` | 规划请求已持久化 |
| `PLANNING -> WAITING_APPROVAL` | 合法计划已生成且策略要求审批 |
| `PLANNING -> SCHEDULED` | 合法计划已生成且策略允许免审批 |
| `WAITING_APPROVAL -> SCHEDULED` | 当前计划快照审批通过 |
| `SCHEDULED -> DISPATCHED` | 分配和执行命令已持久化 |
| `DISPATCHED -> RUNNING` | 边缘确认至少一个首要步骤实际开始 |
| `RUNNING -> PAUSED` | 必要运行步骤均已确认安全暂停 |
| `RUNNING -> COMPLETED` | 所有必要步骤成功且结果校验通过 |
| 任意可执行态 -> `EMERGENCY_STOPPED` | 边缘或云端收到已验证急停事实 |

注意：

- “取消请求已发送”不等于 `CANCELLED`；需要边缘停止确认，无法确认时进入失败/人工处置流程并保持真实状态。
- 拒绝审批使用 `CANCELLED + APPROVAL_REJECTED`，不新增 `REJECTED` Mission 状态。
- `EMERGENCY_STOPPED` 不可恢复；复位后创建新 Mission。

### 6.6 Mission Step 状态 `SM-MISSION-002`

```text
PENDING -> READY
READY -> DISPATCHED | CANCELLED
DISPATCHED -> VALIDATING | FAILED | CANCELLED | EMERGENCY_STOPPED
VALIDATING -> RUNNING | FAILED | CANCELLED | EMERGENCY_STOPPED
RUNNING -> PAUSED | SUCCEEDED | FAILED | TIMEOUT | CANCELLED | EMERGENCY_STOPPED
PAUSED -> RUNNING | FAILED | CANCELLED | EMERGENCY_STOPPED
```

终态：

```text
SUCCEEDED
FAILED
TIMEOUT
CANCELLED
EMERGENCY_STOPPED
```

- 依赖步骤全部 `SUCCEEDED` 后才可 `PENDING -> READY`。
- 可选步骤失败是否导致 Mission 失败由计划明确声明。
- `SUCCEEDED` 必须来自边缘执行结果和输出 Schema 校验，不能由超时推断。
- 步骤重试是新的 `attempt_no`/`execution_id`，不覆盖旧尝试。

### 6.7 Approval 状态 `SM-APPROVAL-001`

```text
PENDING -> APPROVED | REJECTED | EXPIRED | INVALIDATED | CANCELLED
```

所有终态不可再修改：

- `APPROVED`：当前绑定快照已通过。
- `REJECTED`：审批人明确拒绝。
- `EXPIRED`：超过审批截止时间。
- `INVALIDATED`：计划、地图、策略、机器人或风险快照变化。
- `CANCELLED`：Mission 被取消。

重新审批创建新 Approval，不复用旧记录。

### 6.8 Skill Version 状态 `SM-SKILL-001`

```text
DRAFT -> VALIDATING
VALIDATING -> DRAFT | APPROVED
APPROVED -> PUBLISHED | DRAFT
PUBLISHED -> DEPRECATED
DEPRECATED -> RETIRED
```

- `PUBLISHED`、`DEPRECATED`、`RETIRED` 内容不可编辑。
- 验证失败回到 `DRAFT` 并保留失败证据。
- 同一 Skill 可以有多个已发布 major，默认版本必须显式。
- `RETIRED` 不再允许新任务引用，但历史 Trace 必须可解析。

### 6.9 Policy Version 状态 `SM-POLICY-001`

```text
DRAFT -> VALIDATING
VALIDATING -> DRAFT | APPROVED
APPROVED -> PUBLISHED
PUBLISHED -> SUPERSEDED | REVOKED
```

- `REVOKED` 用于安全紧急撤销，不删除版本。
- `SUPERSEDED` 仍用于历史任务回放。
- 边缘缓存必须报告生效版本；未知/过期对运动命令 fail closed。

### 6.10 MCP Tool Version 状态 `SM-MCP-001`

```text
DRAFT -> VALIDATING -> APPROVED -> PUBLISHED -> DEPRECATED -> RETIRED
```

Tool 独立维护 `rollout_state`：

```text
DISABLED -> CANARY -> ENABLED
CANARY | ENABLED -> DISABLED
```

发布状态表示契约生命周期，灰度状态表示运行可用范围，二者禁止混成一个字段。

### 6.11 Edge Execution 状态 `SM-EXEC-001`

```text
PENDING -> VALIDATING
VALIDATING -> RUNNING | FAILED | CANCELLED | EMERGENCY_STOPPED
RUNNING -> PAUSED | SUCCEEDED | FAILED | TIMEOUT | CANCELLED | EMERGENCY_STOPPED
PAUSED -> RUNNING | FAILED | CANCELLED | EMERGENCY_STOPPED
```

- 进程重启后无法确认物理事实时，本地恢复标记为 `UNKNOWN`，先重建设备状态；不能把云端步骤改为成功。
- 重复 `command_id` 返回首次执行的状态/结果。
- `TIMEOUT` 前必须执行 Skill 的安全停止/清理，并记录清理结果。

### 6.12 控制租约状态 `SM-CONTROL-001`

```text
REQUESTED -> ACTIVE | REJECTED
ACTIVE -> RELEASED | EXPIRED | REVOKED | LOST
```

- `ACTIVE` 必须具有唯一 fencing token。
- `REVOKED`：高优先级任务、急停、管理员或安全策略撤销。
- `LOST`：链路丢失且边缘触发本地超时释放。
- 所有终态不可恢复；重新接管创建新租约。

### 6.13 告警状态 `SM-ALARM-001`

告警主状态：

```text
OPEN -> ACKNOWLEDGED -> RESOLVED
OPEN -> RESOLVED
ACKNOWLEDGED -> OPEN
RESOLVED -> OPEN
```

抑制是正交维度：

```text
NOT_SUPPRESSED <-> SUPPRESSED
```

- 同一告警去重键再次出现时，`RESOLVED -> OPEN` 并增加 occurrence。
- `ACKNOWLEDGED` 不改变问题是否存在。
- 抑制只影响通知，不丢弃事件，也不自动解决告警。

### 6.14 Media Asset 状态 `SM-MEDIA-001`

```text
PENDING_UPLOAD -> UPLOADED -> VERIFYING
VERIFYING -> AVAILABLE | QUARANTINED | FAILED
AVAILABLE -> EXPIRED | DELETED
QUARANTINED -> AVAILABLE | DELETED
```

- 只有 `AVAILABLE` 可被普通业务引用。
- `DELETED` 是元数据保留的逻辑终态，对象删除结果需记录。
- 审计证据保留策略可禁止普通删除。

### 6.15 OTA 状态

#### 发布状态 `SM-OTA-001`

```text
DRAFT -> VERIFYING -> READY
VERIFYING -> REJECTED
READY -> DEPRECATED | REVOKED
```

#### 部署状态 `SM-OTA-002`

```text
CREATED -> SCHEDULED -> RUNNING
RUNNING -> PAUSED | SUCCEEDED | FAILED | ROLLING_BACK
PAUSED -> RUNNING | CANCELLED | ROLLING_BACK
ROLLING_BACK -> ROLLED_BACK | FAILED
```

每台设备还有独立 deployment item 状态，聚合状态不能覆盖单设备证据。

### 6.16 Trace 和 Memory 状态

Trace `SM-TRACE-001`：

```text
OPEN -> FINALIZING -> COMPLETE
OPEN | FINALIZING -> INCOMPLETE
```

- `INCOMPLETE` 表示存在证据缺口，不得伪造成 `COMPLETE`。
- 后续补传可生成新索引修订，但原始事件不修改。

Memory `SM-MEMORY-001`：

```text
CANDIDATE -> ACTIVE | REJECTED
ACTIVE -> SUPERSEDED | INVALIDATED
```

Improvement `SM-IMPROVE-001`：

```text
DRAFT -> EVALUATING
EVALUATING -> DRAFT | WAITING_APPROVAL
WAITING_APPROVAL -> APPROVED | REJECTED | INVALIDATED
APPROVED -> CANARY -> RELEASED
CANARY -> ROLLED_BACK | RELEASED
RELEASED -> ROLLED_BACK | SUPERSEDED
```

任何改进状态都不能直接改变机器人运行状态。

### 6.17 公共版本和异步 Operation 状态

版本化配置 `SM-VERSIONED-CONFIG`：

```text
DRAFT -> VALIDATING
VALIDATING -> DRAFT | PUBLISHED
PUBLISHED -> SUPERSEDED | REVOKED
```

通用异步操作 `SM-ASYNC-OPERATION`：

```text
PENDING -> RUNNING | CANCELLED
RUNNING -> SUCCEEDED | FAILED | CANCELLED
```

- Operation 进度是 0-100 的投影，不替代状态。
- 重试创建新 attempt 并保留旧失败记录。
- `SUCCEEDED` 必须带结果引用；`FAILED` 必须带稳定原因码。

### 6.18 Fleet 资源状态

资源锁 `SM-RESOURCE-LOCK`：

```text
REQUESTED -> ACTIVE | REJECTED
ACTIVE -> RELEASED | EXPIRED | REVOKED
```

Assignment `SM-FLEET-ASSIGNMENT`：

```text
PROPOSED -> RESERVED | REJECTED
RESERVED -> DISPATCHED | CANCELLED | EXPIRED
DISPATCHED -> ACTIVE | FAILED | CANCELLED
ACTIVE -> COMPLETED | FAILED | CANCELLED
```

- 锁终态不可恢复，续租更新租约时间和 fencing 世代但不重用已过期锁。
- Assignment 完成不自动推断 Mission 完成，由 Mission 聚合步骤事实。

### 6.19 地图版本和服务健康状态

地图版本 `SM-MAP-VERSION`：

```text
DRAFT -> VALIDATING
VALIDATING -> DRAFT | APPROVED
APPROVED -> PUBLISHED
PUBLISHED -> SUPERSEDED | REVOKED
```

服务健康 `SM-SERVICE-HEALTH`：

```text
UNKNOWN -> STARTING
STARTING -> HEALTHY | DEGRADED | UNHEALTHY
HEALTHY -> DEGRADED | UNHEALTHY | STOPPED
DEGRADED -> HEALTHY | UNHEALTHY | STOPPED
UNHEALTHY -> STARTING | STOPPED
STOPPED -> STARTING
```

健康是观测投影，不提供人工“设为 HEALTHY”接口。

### 6.20 备份和恢复状态

备份 `SM-BACKUP-OPERATION`：

```text
PENDING -> RUNNING | CANCELLED
RUNNING -> VERIFYING | FAILED | CANCELLED
VERIFYING -> SUCCEEDED | FAILED
```

恢复演练 `SM-RESTORE-OPERATION`：

```text
PLANNED -> RESTORING | CANCELLED
RESTORING -> VERIFYING | FAILED
VERIFYING -> SUCCEEDED | FAILED
```

备份只有 `SUCCEEDED` 且带验证摘要才可作为恢复点；恢复演练失败必须产生告警和整改项。

## 7. 核心业务流程与事务边界

### 7.1 登录、授权和撤权

```text
用户登录
-> IAM 校验状态/凭据/锁定
-> 创建可撤销会话
-> 返回短期 Access Token
-> 前端加载 /me、权限和数据范围
-> 页面查询/Command 再次由后端鉴权
```

撤权流程：

```text
角色/权限变更事务
-> 写审计 + Outbox
-> 发布 authorization.changed
-> 清理授权缓存
-> 受影响 WebSocket/会话重新校验
-> 后续请求立即使用新权限
```

前端缓存权限不具备授权效力。

### 7.2 网关和机器人注册

```text
网关用注册令牌 + mTLS 发起
-> 校验身份/硬件/版本
-> 建立 PENDING 网关
-> 管理员审批 ACTIVE
-> 网关上传机器人 Manifest
-> Robot Registry 校验型号/能力/摘要
-> 建立 PENDING Robot
-> 绑定网关/策略/地图
-> 激活
-> 心跳租约计算 ONLINE
```

事务边界：

- 网关/机器人主数据、审计、Outbox 同一事务。
- 证书签发、对象存储和远程握手在事务外，通过 Operation 记录补偿。
- Manifest 更新不得覆盖正在执行任务所绑定的 Manifest 快照。

### 7.3 自然语言任务创建与规划

```text
POST Mission（幂等）
-> CREATED
-> Outbox: planning_requested
-> Worker: PLANNING
-> 调用 QwenPaw Provider
-> 校验 PlanProposal Schema
-> Planner 解析 Skill/地图/参数/能力
-> DAG/风险/补偿/可行性
-> 保存 Plan Revision
-> Policy 任务级评估
-> WAITING_APPROVAL 或 SCHEDULED
```

关键失败：

- Agent 超时、模型不可用、非法 Schema：`FAILED`，保留错误和原始输出摘要。
- 信息不足：规划结果返回澄清需求，不进入执行状态；通过新的用户输入继续同一规划修订。
- 用户在规划期间取消：迟到的 Agent 结果记录后丢弃，不复活任务。

### 7.4 审批和失效

```text
创建 Approval（绑定 plan/map/policy/robot revision）
-> PENDING
-> 审批人查看完整快照和风险
-> APPROVED / REJECTED
-> APPROVED 时原子校验快照仍有效
-> Mission 进入 SCHEDULED
```

计划或依赖版本变化时：

```text
变更事件
-> 找到 PENDING/APPROVED 但尚未执行的 Approval
-> INVALIDATED
-> Mission 回到需要重新评估的受控流程
```

不得继续使用“曾经批准过”的布尔字段。

### 7.5 调度、下发与开始执行

```text
SCHEDULED
-> 构造候选并硬过滤
-> 记录评分
-> 原子预占资源 + fencing token
-> 创建 Assignment
-> 生成签名 command_id
-> Mission DISPATCHED
-> Outbox 下发
-> Edge 验签/防重放/状态检查
-> Safety Gateway
-> Execution VALIDATING/RUNNING
-> 边缘确认
-> Mission RUNNING
```

- NATS/HTTP 发送失败不回滚已提交数据库事实，由 Outbox 重试。
- 边缘拒绝必须包含原因并释放或保留资源（按原因策略）。
- 下发超时不能假定边缘未执行；需按 `command_id` 查询/对账。

### 7.6 正常完成

```text
Edge Skill 执行
-> 输出 Schema 校验
-> execution SUCCEEDED
-> step SUCCEEDED
-> 解锁依赖步骤
-> 所有必要步骤 SUCCEEDED
-> Mission COMPLETED
-> 释放资源
-> Trace FINALIZING/COMPLETE
-> Memory 提取异步开始
```

Mission 完成、资源释放和完成事件在云端事实事务中保持一致；Memory 失败不能反向把已完成任务改为失败。

### 7.7 暂停、恢复和取消

暂停：

```text
检查 RUNNING + 权限 + Skill 可暂停
-> pause_requested
-> Edge 安全暂停
-> 确认后 Step/Mission PAUSED
```

恢复：

```text
检查 PAUSED
-> 重新校验权限/审批有效性/策略/机器人/环境/资源
-> Edge Safety 再检查
-> 确认后 RUNNING
```

取消：

```text
标记 cancel_requested（不提前写 CANCELLED）
-> Edge 停止/清理
-> 确认各活动步骤终止
-> Mission CANCELLED
-> 释放资源并完成 Trace
```

### 7.8 急停和复位

触发源：

- 本地物理按钮。
- Edge Safety 自动触发。
- 有权限用户从云端触发。

执行：

```text
最高优先级急停
-> Safety 锁存
-> 中断 Executor
-> Adapter 发送 stop/damping/zero velocity
-> Robot safety_state ESTOPPED
-> 活动 execution/step/mission EMERGENCY_STOPPED
-> 告警 OPEN
-> 事件本地持久化并回传
```

云断连时从“本地持久化”开始执行，不能等云端确认。

复位：

```text
独立复位权限 + 原因 + 二次确认
-> 现场条件/本体/环境检查
-> ESTOPPED -> RESTRICTED
-> 完整自检
-> RESTRICTED -> NORMAL
-> 告警满足条件后 RESOLVED
```

原 Mission 保持终态，不恢复。

### 7.9 弱网、离线和重连对账

离线：

- Edge 记录最后云连接时间并进入离线模式。
- 未声明可离线完成的运动 Skill 默认安全中止。
- 急停、Watchdog、本地策略和日志缓存继续工作。
- 本地事件按优先级持久化；不得因遥测堆积挤掉安全事件。

重连顺序：

```text
身份/证书握手
-> 时间和版本校准
-> 上传急停/执行终态/安全事件
-> 按 command_id 对账
-> 补 Mission/Step 事实
-> 补普通 Trace/告警
-> 补遥测/媒体
-> 恢复新命令接收
```

云端不得用旧快照覆盖边缘更晚的物理终态。

### 7.10 多机器人故障转移

```text
检测原机器人故障/离线
-> 暂停后继步骤
-> 确认原执行停止或 fencing 失效
-> 检查 Skill 是否可重分配及副作用
-> 新候选/评分/审批策略
-> 新 Assignment 和 command_id
-> 继续或生成补偿
```

无法证明原执行停止时禁止自动重分配有物理副作用的步骤。

### 7.11 失败经验到生产改进

```text
失败 Mission/Step/Trace
-> 提取 Failure Case
-> 人工确认原因/证据
-> 生成 Skill/Policy 新草稿
-> 历史回放测试
-> 仿真评测
-> 安全审批
-> Canary
-> 指标对比
-> RELEASED 或 ROLLED_BACK
```

Memory 或 Agent 只能提出候选，不能写已发布版本。

## 8. 数据一致性和状态落库规则

### 8.1 聚合与并发

| 聚合 | 聚合根 | 并发控制 |
| --- | --- | --- |
| User | `user` | 用户版本；凭据/会话独立安全事务 |
| Role | `role` | 角色版本；授权变更事件失效缓存 |
| Robot | `robot` | 主数据版本；高频状态使用独立投影版本 |
| Skill | `skill_version` | 不可变发布版本；草稿乐观锁 |
| Mission | `mission` | `aggregate_version` 串行状态迁移 |
| Approval | `approval_record` | 单次决定唯一约束 |
| Resource Lock | `resource_lock` | 唯一资源键 + 租约 + fencing token |
| Policy | `policy_version` | 不可变发布版本 |
| Alarm | `alarm` | 去重键 + 乐观锁 |
| OTA Deployment | `ota_deployment` | 聚合版本；设备项独立版本 |

- Command 必须携带或由服务端确定 `expected_version`。
- 状态更新 SQL 必须包含旧状态/版本条件；影响行数不为 1 即并发冲突。
- 冲突返回 409 和最新版本引用，不做最后写入获胜。

### 8.2 事实表、历史表和投影

```text
事实/聚合表：当前权威状态和版本
事件/历史表：只追加的状态变化证据
查询投影：为页面/统计优化，可由事件重建
遥测表：设备观察值，不直接等于业务状态
```

示例：

- `robot` 保存注册主数据。
- `robot_runtime_projection` 保存连接/运行/安全当前投影。
- `robot_state_event` 只追加每次状态变化。
- `telemetry` 保存电量、位姿等观察值。

禁止仅更新当前状态而不写状态事件，也禁止前端根据遥测直接写业务状态。

### 8.3 状态事件统一字段

所有状态历史至少包含：

```text
event_id
aggregate_type
aggregate_id
aggregate_version
state_machine_id
state_dimension
from_state
to_state
reason_code
reason_detail
actor_type
actor_id
occurred_at
recorded_at
trace_id
causation_id
payload_snapshot
```

状态事件与聚合变更、审计、Outbox 必须在同一数据库事务。

### 8.4 设备时间、乱序和新鲜度

- 设备事件带 `observed_at`、`sent_at`、`received_at`、设备序列号和启动 epoch。
- 同一启动 epoch 内按序列号排序；设备重启产生新 epoch。
- 迟到遥测可以入历史，但不能覆盖更新的当前投影。
- 连接状态只由云端租约计算；设备上报“我在线”不是权威状态。
- 位姿、电量、传感器状态必须返回 `age_ms`/采集时间；超过策略阈值视为未知或过期。

### 8.5 状态原因码

每个失败/阻断/迁移必须使用稳定原因码，例如：

```text
AUTH_ACCOUNT_LOCKED
ROBOT_HEARTBEAT_EXPIRED
ROBOT_NOT_SCHEDULABLE
MISSION_PLAN_SCHEMA_INVALID
MISSION_INVALID_TRANSITION
APPROVAL_SNAPSHOT_CHANGED
FLEET_RESOURCE_CONFLICT
SAFETY_SPEED_LIMIT_EXCEEDED
SAFETY_RESTRICTED_AREA
SAFETY_ESTOP_LATCHED
EDGE_COMMAND_EXPIRED
EXECUTION_OUTPUT_INVALID
MEDIA_DIGEST_MISMATCH
```

- 原因码定义在契约中，不能从异常类名临时生成。
- 自然语言通过 `message_key + arguments` 生成。
- 数据库保存原因码和必要结构化参数，不只保存一段中文。

### 8.6 数据删除和保留

- 用户、机器人、Skill、Policy 以停用/退役为主，不因被引用而物理删除。
- Mission、状态历史、审批、Safety、审计和 Trace 事实不可普通删除。
- 遥测、媒体和日志按分类保留策略归档/删除。
- 删除对象存储资产先验证法律/审计保留，再产生可审计删除 Operation。
- 隐私匿名化保留业务证据但移除可识别信息，并记录处理依据。

### 8.7 数据库约束必须表达不变量

除领域代码外，数据库至少使用：

- 唯一约束：public ID、权限码、字典 code、版本号、幂等键、事件 ID。
- Check：百分比/电量范围、非负时长、状态值、时间顺序。
- 部分唯一索引：一个资源最多一个 ACTIVE 控制租约/资源锁。
- 外键：同一领域内的强一致引用。
- 不可为空：业务不变量字段。

禁止把所有有效性都寄托在前端或 Bean Validation。

### 9.2 查询响应中的版本和动作

所有聚合详情响应至少包含：

```json

  "links": {}

```

列表项可使用精简模型，但状态维度和数据时间不得丢失。

### 9.3 WebSocket 频道

统一连接：

```text
/ws/v1/events
```

客户端发送订阅：

```json
{
  "type": "subscribe",
  "subscription_id": "sub-001",
  "topics": [
    "mission:mission-001",
    "robot:dog-001",
    "alarms:scope"
  ],
  "cursor": "event-previous"
}
```

服务端消息：

```json
{
  "type": "event",
  "subscription_id": "sub-001",
  "event_id": "01...",
  "aggregate_id": "mission-001",
  "aggregate_version": 8,
  "event_type": "mission.running",
  "occurred_at": "2026-07-03T08:02:00Z",
  "data": {}
}
```

规则：

- 订阅主题由服务端解析并做资源范围鉴权。
- 心跳、重鉴权、限流、慢消费者和最大订阅数必须定义。
- 游标不可补齐时返回 `snapshot_required`，前端重新取 REST 快照。

### 9.4 MCP Tool 契约

MCP Tool 输入除了业务参数，还必须由网关注入不可伪造的调用上下文：

```json
{
  "delegated_actor_id": "user-001",
  "session_id": "session-001",
  "trace_id": "01...",
  "tool_version": "1.0.0"
}
```

Agent 提供的同名字段必须被忽略/拒绝。Tool 返回受理事实、资源链接和稳定错误，不返回内部堆栈。

### 9.5 事件兼容

- 聚合事件包含 `aggregate_version`，消费者按聚合有序协调。
- 投影消费者发现版本跳跃时暂停该聚合并补事件/重建，不静默跳过。
- 所有 `*.changed` 事件必须说明是事实变化还是投影变化；领域集成只消费事实事件。
- WebSocket 可以使用面向 UI 的投影事件，但不得反向驱动领域事实。

## 11. AI 实施顺序

### 11.1 每个功能的纵向实施

AI 必须按以下顺序完成一个功能，再进入同阶段下一个功能：

1. 确认更早阶段已完成，读取功能 profile、部署目标和外部契约；外部契约未 `PINNED` 时先锁定，不得开始集成。
2. 在机器清单把功能标记为 `IN_PROGRESS`，不得提前 `DONE`。
3. 写用户场景、权限、状态迁移和失败场景。
4. 新增/修改 OpenAPI、AsyncAPI、Schema 和错误码。
5. 写 Flyway migration、约束和种子数据。
6. 实现领域聚合、状态机和单元测试。
7. 实现应用 Command/Query、审计、Outbox/Inbox。
8. 实现协议适配和集成测试。
9. 实现前端路由、页面状态、权限、i18n 和组件测试。
10. 实现 WebSocket/实时对账。
11. 完成 E2E/仿真/适用的 HIL 证据。
12. 在全部 `deployment_targets` 验证配置、迁移、健康、升级和回滚，接入指标/日志/Trace/告警。
13. 生成非空的测试、安全、部署、可观测证据并更新 Runbook；全部门禁通过后标记 `DONE`。

### 11.2 禁止横向铺空壳

以下不算进度：

- 创建所有模块但只有空 Controller/Service/Mapper。
- 创建全部菜单但页面只有“开发中”。
- 先建几十张表但没有聚合规则和迁移测试。
- 为每个状态生成 CRUD 修改接口。
- 用固定 JSON 或 Mock Robot 贯通生产代码路径。

正确方式是按 M0-M6 完成可运行纵向切片。

### 11.3 第一条业务纵向切片

M0/M1 后的第一条切片固定为：

```text
登录
-> 查看仿真机器人
-> 创建受限 move_forward Mission
-> QwenPaw/结构化计划
-> Policy/审批
-> 调度仿真机器人
-> Edge Safety 限速
-> Local Skill Executor
-> F-ADAPTER-001 ROS2 Simulation Adapter
-> 状态/Trace/告警回传
-> 前端查看完成结果
-> 急停专项路径
```

该切片同时验证公共能力、契约、状态机、云边链路和可观测性。

F-ADAPTER-001 属于 M2，只验证 ROS2/仿真主路径，不以真实硬件 HIL 为完成前提。F-ADAPTER-002 属于 M3，承载 ROS1、Unitree 和自研协议兼容及其受控 HIL，禁止用 M2 仿真报告替代。

## 12. 完成审计

AI 声称功能完成前必须提供当前工作树中的直接证据：

| 审计问题 | 合格证据 |
| --- | --- |
| 页面是否完整 | 路由、页面、组件测试、各视图状态截图/E2E |
| 后端是否完整 | 用例实现、领域单测、权限和状态迁移测试 |
| 数据是否完整 | Migration、约束、空库/升级测试、所有权检查 |
| 实时是否一致 | 快照+事件对账、重复/乱序/断线测试 |
| 安全是否闭环 | Policy/Safety/急停/防重放测试和 Trace |
| 是否复用公共能力 | 依赖/架构测试，无重复用户/权限/字典/i18n |
| 是否可运行 | 统一脚本从空环境启动和健康检查 |
| 是否可部署回滚 | 每个 deployment target 的配置、迁移、升级、回滚和健康证据 |
| 是否可运维 | 指标、日志、Trace、告警规则、仪表盘和 Runbook 的联动证据 |
| 是否经过安全门禁 | 非空安全报告、Secret/SAST/依赖/许可证/镜像扫描及适用的仿真/HIL |
| 是否可追溯 | 功能 ID、用例 ID、状态机 ID、验收 ID 可关联 |
| 是否真实完成 | 无 TODO/Mock/跳过测试；未做 HIL 明确标记 |

仅有源码、编译成功或接口返回 200 都不足以证明完整实施。

机器校验：

```text
python scripts/validate_platform_manifest.py
python scripts/validate_platform_manifest.py --require-complete
```

第一条用于日常检查 Schema、双向 ID 追踪、架构/公共能力归属、阶段门禁、外部契约和 C01-C24 覆盖；第二条是全平台完成门禁，要求全部基线功能为 `DONE`。完成证据必须位于规定目录且包含非空文件；只有路径存在、空文件或空目录均不合格。
